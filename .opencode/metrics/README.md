# 数据反馈闭环 — Metrics & Continuous Improvement

## 核心思想

> **没有度量就没有改进。**  
> 每次 AI 操作都会记录到 changelog，定期分析这些数据，找出流程瓶颈，然后改进流程。  
> 改进后继续收集数据，看改进是否有效。

## 闭环流程

```
                      ┌─────────────────────────┐
                      │   AI 执行任务             │
                      │   (自动记录 changelog)    │
                      └──────────┬──────────────┘
                                 │
                                 ▼
                      ┌─────────────────────────┐
                      │   收集数据               │
                      │   collect.ps1 分析       │
                      │   changelog JSONL        │
                      └──────────┬──────────────┘
                                 │
                                 ▼
                      ┌─────────────────────────┐
                      │   生成报告               │
                      │   首次通过率/失败模式/    │
                      │   瓶颈分析               │
                      └──────────┬──────────────┘
                                 │
                                 ▼
                      ┌─────────────────────────┐
                      │   识别改进点             │
                      │   Prompt 不够好?         │
                      │   任务拆分太粗?          │
                      │   测试覆盖不够?          │
                      └──────────┬──────────────┘
                                 │
                                 ▼
                      ┌─────────────────────────┐
                      │   改进流程               │
                      │   更新 prompt            │
                      │   补充脚本/测试          │
                      │   调整任务拆分粒度        │
                      └──────────┬──────────────┘
                                 │
                                 └──→ 回到第一步（循环）
```

## 关键指标

| 指标 | 目标 | 怎么算 | 反映什么 |
|------|------|--------|---------|
| **首次通过率** | > 60% | 一次提交即通过的 task 占比 | Prompt 质量、任务拆分合理性 |
| **平均修改轮次** | < 3 | 每个 task 平均需要几次提交 | 验收标准是否清晰 |
| **最常见失败模式** | 持续下降 | 哪个阶段（编译/测试/审查）失败最多 | 流程瓶颈在哪 |
| **恢复时间** | < 15 分钟 | 从紧急冻结到恢复健康的耗时 | 紧急预案有效性 |

## 使用方式

### 手动收集（开发过程中）

```powershell
# 查看今天的变更记录
Get-Content .opencode\changelog\2026-07-05.jsonl | ConvertFrom-Json

# 查看特定 agent 的活动
Get-Content .opencode\changelog\*.jsonl | ConvertFrom-Json | Where-Object { $_.agent -eq "build-1" }
```

### 自动生成报告

```powershell
# 分析最近 7 天的数据
.opencode\metrics\collect.ps1 -Days 7

# 分析最近 1 天的数据
.opencode\metrics\collect.ps1 -Days 1

# 指定报告输出位置
.opencode\metrics\collect.ps1 -Days 14 -ReportPath "reports/sprint-review.md"
```

报告会包含：
- 总体统计（总任务数、完成数、失败率）
- 首次通过率
- 最常见的失败原因
- 最常修改的文件（热点分析）
- Agent 活动统计
- **自动改进建议**（基于数据给出可操作的建议）

### 定期执行

建议每周跑一次：

```powershell
# 创建 reports 目录
mkdir reports -Force

# 生成周报
.opencode\metrics\collect.ps1 -Days 7 -ReportPath "reports/week-$(Get-Date -Format 'yyyy-MM-dd').md"
```

## 改进流程详解

拿到报告后，按以下顺序处理：

### 1. 看首次通过率

| 首次通过率 | 含义 | 改什么 |
|:----------:|------|--------|
| < 40% | ❌ 严重 | Prompt 模板太模糊，任务拆分太粗 |
| 40-60% | ⚠️ 偏低 | 需要补充更多约束条件 |
| 60-80% | ✅ 达标 | 持续优化 |
| > 80% | 🏆 优秀 | 保持 |

### 2. 看失败模式

发现某个环节（比如编译检查）失败特别多时：

```
编译失败多 → 是不是 LSP 没开？→ 检查 opencode.json 的 "lsp": true
测试失败多 → 是不是测试用例不够准？→ 更新 test-generation.md
审查失败多 → 是不是 prompt 遗漏了约束？→ 更新相关 prompt
```

### 3. 看最常修改的文件

```
文件 X 被改了 10 次 → 这个文件是不是太复杂了？→ 考虑重构
文件 Y 被连续改了 3 次 → 上次没改好？→ 检查回归测试
```

### 4. 应用改进

```
找到瓶颈 → 改对应的文件 → 记录改了什么 → 下周看数据是否改善
```

## 数据从哪里来

所有的数据来自 `.opencode/changelog/` 目录下的 JSONL 文件：

```
.opencode/changelog/
├── 2026-07-05.jsonl    ← 每天的变更记录
├── 2026-07-06.jsonl
└── ...
```

每条记录是 AI 操作的结构化日志：

```json
{"id":"chg_12345678","agent":"build-1","task":"实现登录","action":"create","files":["login.cs"],"status":"passed","timestamp":"2026-07-05T14:30:00Z"}
```

**重要**：changelog 是 append-only（只能追加，不能修改删除），保证数据不可篡改。

## 质量门禁改进清单

每轮改进后检查：

- [ ] 首次通过率比上轮提升了吗？
- [ ] 最常见的失败模式减少了吗？
- [ ] 改进措施记录到文档了吗？
- [ ] 对应的 prompt/脚本/测试更新了吗？
